2. ARQUITECTURA MULTI-TENANT
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

El sistema opera con aislamiento total de datos entre prestamistas (tenants).
Cada empresa prestamista es un "tenant" con sus propios datos, usuarios,
configuraciones y plan de suscripción.

NIVELES DE USUARIO:
  1. platform_owner   — Dueño de PrestaMax. Acceso total al panel de administración.
                        Puede crear/gestionar tenants, planes, usuarios globales,
                        ver auditorías, hacer backups, borrar datos por empresa.
  2. platform_admin   — Administrador de plataforma. Similar al owner pero sin
                        algunas acciones destructivas.
  3. platform_support — Soporte técnico de la plataforma.
  4. tenant_owner     — Dueño de una empresa prestamista. Acceso total dentro de
                        su tenant. Puede invitar cobradores y gestionar su negocio.
  5. admin            — Administrador dentro de un tenant. Acceso completo.
  6. prestamista      — Oficial de préstamos. Puede crear, aprobar, desembolsar.
  7. oficial          — Igual que prestamista (alias).
  8. cashier          — Cajero. Principalmente registra pagos y recibos.
  9. cobrador         — Cobrador. Ve préstamos asignados, registra pagos, cobros.

UN USUARIO PUEDE PERTENECER A MÚLTIPLES TENANTS (empresas), cada membresía
tiene sus propios roles y permisos dentro de cada tenant.

CABECERAS HTTP REQUERIDAS:
  - Authorization: Bearer <JWT_TOKEN>
  - x-tenant-id: <UUID del tenant>  (requerido en todas las rutas tenant-scoped)

VERIFICACIÓN DE SUSCRIPCIÓN:
  El middleware requireTenant verifica automáticamente que la suscripción del
  tenant no haya expirado. Si expiró, responde HTTP 402 con:
  { error: "Tu suscripción ha expirado...", code: "SUBSCRIPTION_EXPIRED" }
  El frontend debe redirigir a /billing para que el usuario pague.

5. BASE DE DATOS — TABLAS PRINCIPALES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

TABLAS Y SU PROPÓSITO (29 tablas):

  plans                   — Planes de suscripción (Trial, Starter, Básico,
                            Profesional, Enterprise). Precio, límites, features
                            como JSON array de PermKeys, trial_days (default 10).

  users                   — Usuarios globales del sistema. platform_role define
                            si son dueño/admin/soporte de la plataforma o "none".

  tenants                 — Empresas prestamistas. Cada una tiene su plan,
                            estado de suscripción, configuración de moneda,
                            token público para el portal de solicitudes,
                            stripe_customer_id, stripe_subscription_id,
                            subscription_start, subscription_end,
                            datos notariales para contratos legales.

  tenant_settings         — Configuración avanzada por tenant: tasa de mora diaria,
                            días de gracia, base de mora, tipo de rebate, pesos
                            del score crediticio, configuración WhatsApp,
                            multi-moneda habilitado, monedas habilitadas.

  tenant_memberships      — Relación usuario-empresa. Un usuario puede tener
                            membresías en múltiples empresas. Cada membresía
                            almacena roles (JSON array) y permisos explícitos
                            (JSON object key:boolean).

  branches                — Sucursales de cada empresa prestamista.

  clients                 — Clientes de préstamos. Datos personales, laborales,
                            referencias, score crediticio (1-5), consentimientos.

  client_references       — Referencias personales/laborales del cliente.
  guarantors              — Fiadores (personas que garantizan préstamos).
  client_documents        — Documentos adjuntos del cliente (cédula, comprobantes).

  loan_products           — Productos de préstamo configurables: tipo, tasas,
                            plazos, frecuencias, amortización, comisiones,
                            si requiere garantía, política de rebate.

  loans                   — Préstamos. Ciclo de vida completo:
                            draft → pending_approval → approved → disbursed →
                            active → liquidated / in_mora / write_off / voided.

  installments            — Cuotas del préstamo. Generadas al desembolsar.
                            Estado: pending/partial/paid/overdue/waived.

  loan_guarantors         — Relación préstamo-fiador (M:M).
  loan_guarantees         — Garantías en especie del préstamo.
  guarantee_categories    — Categorías de garantías por tenant.

  payments                — Pagos registrados. Distribución (mora, cargos,
                            interés, capital), método, cuenta bancaria destino,
                            tipo (regular/interest_only/prorroga/prepago), void.

  payment_items           — Detalle por cuota de cómo se distribuyó cada pago.

  receipts                — Recibos emitidos por cada pago.
  receipt_series          — Series numéricas de recibos por tenant.

  contracts               — Contratos generados. Firma: physical/digital.
  contract_templates      — Plantillas con variables {{client_name}}, etc.

  whatsapp_templates      — Plantillas de mensajes WhatsApp por evento.
  whatsapp_messages       — Historial de mensajes (sin envío real implementado).

  bank_accounts           — Cuentas bancarias del prestamista.
  account_transfers       — Transferencias entre cuentas propias.

  income_expenses         — Ingresos y gastos del negocio. Columna fecha:
                            transaction_date (NO "date").

  payment_promises        — Promesas de pago, con seguimiento de visitas.
  collection_notes        — Notas de gestión de cobranza por préstamo.
  collection_tasks        — Tareas asignadas a cobradores.

  notifications           — Notificaciones internas del sistema.

  audit_logs              — Log de auditoría de todas las acciones importantes.

  loan_requests           — Solicitudes del portal público (datos en Base64).

8. TIPOS DE PRÉSTAMO
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

loan_products.type:
  personal / san / commercial / guarantee

TIPOS DE AMORTIZACIÓN:
  fixed_installment, flat_interest, interest_only, declining_balance

TIPOS DE TASA: daily, weekly, biweekly, monthly, annual
FRECUENCIAS DE PAGO: daily, every_2_days, weekly, biweekly, monthly

CICLO DE VIDA DEL PRÉSTAMO:
  draft → pending_approval → approved → disbursed → active
  active → liquidated / in_mora / write_off / voided

CÁLCULO DE MORA:
  mora_daily = base_amount × mora_rate_daily × dias_vencidos
  - mora_rate_daily: default 0.001 (0.1%)
  - mora_grace_days: default 3
  - mora_max_rate: default 0.30 (30%)
  - mora_base: 'cuota_vencida' o 'saldo_capital'
  - mora_fixed_enabled / mora_fixed_amount: monto fijo en vez de %

OVERTIME: recargo si supera maturity_date sin liquidar.
PRORROGA: pago tipo 'prorroga' desplaza cuotas, cobra prorroga_fee.

REBATE (descuento por pago anticipado):
  proportional / partial / fixed_rate / none

SCORE CREDITICIO (1-5): calculado en cada pago/anulación.
  Factores con pesos configurables: puntualidad (40%), ratio pagados (30%),
  antigüedad (20%), sin mora (10%).

IMPORTACIÓN CSV: masivo con campos currency y exchange_rate_to_dop.

16. CIBERSEGURIDAD IMPLEMENTADA
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

JWT (auth.ts):
  - Algoritmo forzado: { algorithms: ['HS256'] }
  - Token de 7 días de expiración
  - JWT_SECRET validado al inicio

Helmet.js:
  - CSP en producción con whitelist (incluye js.stripe.com)
  - HSTS, noSniff, frameguard deny, hidePoweredBy, xssFilter
  - Referrer-Policy: strict-origin-when-cross-origin

CORS:
  - Whitelist: FRONTEND_URL, prestamax.com, www/app subdominios
  - Headers permitidos: Authorization, X-Tenant-Id, Stripe-Signature

Rate Limiting:
  - Global: 300 req/15min/IP (excepto /health y /api/billing/webhook)
  - Auth: 15 intentos/15min (anti brute-force)
  - Register: 5/hora
  - Admin: 100/15min
  - Bulk import/export: 20/15min

Middleware seguridad: sanitizeInputs, blockKnownBots, detectMaliciousPayload,
                      auditLogger

Aislamiento de Tenants: requireTenant + AND tenant_id=? en todas las queries

Contraseñas: bcrypt cost 12, validación complejidad
             (min 8, 1 mayúscula, 1 número, 1 carácter especial)

ErrorHandler: en producción mensaje genérico (no internals)

/health endpoint: solo { status: "ok" }

Webhook Stripe: firma HMAC verificada via stripe.webhooks.constructEvent
                Body raw preservado con express.raw() ANTES de express.json

PENDIENTES CONOCIDOS:
  - Token blacklist/logout (JWT siguen válidos hasta expirar)
  - Validación MIME type/tamaño en imágenes Base64 portal público
  - Rate limiting individual por usuario autenticado

25. MIGRACIONES DE BASE DE DATOS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

NO existe sistema de migraciones formal. Se usa CREATE TABLE IF NOT EXISTS y
ALTER TABLE ADD COLUMN con try/catch silencioso en initializeDatabase().

Patrón seguro para nuevos deployments:
  - CREATE TABLE IF NOT EXISTS crea solo si no existe
  - ALTER TABLE ADD COLUMN falla silenciosamente si columna ya existe

Al agregar nuevas columnas, usar siempre:
  try { db.exec(`ALTER TABLE tabla ADD COLUMN nueva_col TIPO DEFAULT valor`); } catch(_) {}

[A] MIGRAR STRIPE DE TEST MODE A LIVE MODE
================================================================================
    (Cuando estés listo para cobrar dinero real a prestamistas)

    PASO 1 — ACTIVAR LA CUENTA DE STRIPE PARA LIVE MODE
       URL: https://dashboard.stripe.com/account/onboarding
       Completar todos los datos del negocio:
         - Business name (razón social)
         - Business type (sociedad, persona física, etc.)
         - RNC / Tax ID dominicano
         - Dirección física del negocio
         - Sitio web (opcional pero recomendado)
         - Descripción de productos/servicios
         - Datos del representante legal (cédula, fecha nacimiento)
         - Cuenta bancaria DR para recibir los pagos
         - Verificación de identidad (subir foto de cédula del titular)
       Stripe revisa los datos en 1-3 días hábiles.
       Cuando esté aprobada, el toggle Sandbox/Live arriba a la derecha
       del dashboard estará disponible.

    PASO 2 — CAMBIAR AL MODO LIVE
       En el dashboard de Stripe, click en el toggle arriba a la derecha
       que dice "Sandbox" → cambiarlo a "Live".
       Verificar que la URL ahora dice "/dashboard" SIN "/test/".

    PASO 3 — CREAR LOS 4 PRODUCTOS EN LIVE MODE
       URL: https://dashboard.stripe.com/products (sin /test/)
       Click "+ Add product" y crear UNO POR UNO los mismos 4 productos:

         Producto 1: PrestaMax Starter
           Precio: 29.99 USD, Recurring monthly
         Producto 2: PrestaMax Basico
           Precio: 59.99 USD, Recurring monthly
         Producto 3: PrestaMax Profesional
           Precio: 119.99 USD, Recurring monthly
         Producto 4: PrestaMax Enterprise
           Precio: 249.99 USD, Recurring monthly

       Para cada producto después de crearlo, copiar el Price ID (empieza
       con price_... pero estos serán DIFERENTES a los de test).
       Anotar los 4 Price IDs Live.

    PASO 4 — ACTIVAR EL CUSTOMER PORTAL EN LIVE MODE
       URL: https://dashboard.stripe.com/settings/billing/portal
       Activar las mismas funciones que en test mode:
         - Customers can update payment method
         - Customers can cancel subscriptions
         - Customers can switch plans (seleccionar los 4 productos Live)
         - Customers can update billing address
       Headline: "Gestiona tu suscripcion de PrestaMax"
       Save

    PASO 5 — OBTENER LAS API KEYS LIVE
       URL: https://dashboard.stripe.com/apikeys (sin /test/)
       Copiar:
         - Publishable key (empieza con pk_live_...)
         - Secret key (empieza con sk_live_...) — click "Reveal" primero
       ⚠️ GUARDAR ESTAS KEYS DE FORMA SEGURA — NO commitearlas a Git

    PASO 6 — CREAR EL WEBHOOK EN LIVE MODE
       URL: https://dashboard.stripe.com/workbench/webhooks (sin /test/)
       Click "+ Add destination"

       Step 1 (Select events):
         - Events from: Your account
         - Marcar exactamente los 5 eventos:
           * checkout.session.completed
           * customer.subscription.created
           * customer.subscription.updated
           * customer.subscription.deleted
           * invoice.payment_failed
         - Continue

       Step 2 (Choose destination type):
         - Webhook endpoint → Continue

       Step 3 (Configure your destination):
         - Endpoint URL: https://prestamax-api.onrender.com/api/billing/webhook
         - Description: PrestaMax production live webhook
         - Create destination

       Después de crear, ir a la página del webhook y copiar el "Signing
       secret" (empieza con whsec_...).

    PASO 7 — ACTUALIZAR LAS 6 ENV VARS EN RENDER
       URL: https://dashboard.render.com → servicio prestamax-api → Environment
       EDITAR (no agregar nuevas, EDITAR las existentes):

         STRIPE_SECRET_KEY        = sk_live_... (nueva, paso 5)
         STRIPE_WEBHOOK_SECRET    = whsec_... (nueva, paso 6)
         STRIPE_PRICE_STARTER     = price_... (nuevo Live, paso 3)
         STRIPE_PRICE_BASICO      = price_... (nuevo Live, paso 3)
         STRIPE_PRICE_PROFESIONAL = price_... (nuevo Live, paso 3)
         STRIPE_PRICE_ENTERPRISE  = price_... (nuevo Live, paso 3)

       Click "Save, rebuild, and deploy" abajo.
       Esperar ~3-5 min al redeploy.

    PASO 8 — PROBAR CON UNA TARJETA REAL
       - Refresh el frontend en https://prestamax-umber.vercel.app
       - Crear una cuenta de prueba con un email DIFERENTE al tuyo
         (porque tu cuenta jcpenalo@gmail.com ya está como Enterprise test)
       - Ir a /billing → Suscribirse a Starter ($29.99/mes)
       - Usar una tarjeta REAL personal (NO 4242...)
       - Verificar que se cobra realmente
       - En Stripe dashboard verás el pago en
         https://dashboard.stripe.com/payments
       - Después puedes reembolsar la transacción desde Stripe si era solo
         para pruebas.

    PASO 9 — CONSIDERACIONES IMPORTANTES
       - En Live mode, las transacciones SI cobran dinero real
       - Stripe cobra 2.9% + $0.30 USD por transacción exitosa
       - Stripe transfiere el dinero a tu cuenta bancaria DR cada 2-7 días
         (cronograma se configura en
          https://dashboard.stripe.com/settings/payouts)
       - Mantener actualizado Privacy Policy y Terms con tu información
         legal real (RNC, dirección, etc.)
       - Los productos en TEST mode quedan separados, puedes seguir usando
         test mode para desarrollo

[C] CONVERTIR LA WEB EN APP MOVIL PARA ANDROID Y iOS (CAPACITOR)
================================================================================
    (En espera. Primero estabilizar todas las funciones web)

    Estrategia: usar Capacitor para empaquetar la app React/Vite existente
    en contenedores nativos. Mantiene 100% del codigo actual.

    Costo: Apple Developer $99/ano + Google Play $25 una sola vez.

    PASO 1 — SETUP BASICO (1 dia)
       cd prestamax/frontend
       npm install @capacitor/core @capacitor/cli
       npx cap init prestamax com.digitalconnectdr.prestamax
       npm install @capacitor/android @capacitor/ios
       npx cap add android
       npx cap add ios   (requiere Mac con Xcode)
       npm run build && npx cap sync
       npx cap open android   (abre Android Studio)

    PASO 2 — PULIR PARA MOBILE (1 dia)
       - Verificar responsive en pantallas chicas
       - Splash screen con logo PrestaMax
       - Icon de app (1024x1024 base)
       - Status bar color
       - Orientacion portrait fija
       - Configurar CapacitorConfig en capacitor.config.ts

    PASO 3 — APIs NATIVAS UTILES (opcional, 1-2 dias por feature)
       - Camara nativa para escanear cedulas (cobradores en campo):
           npm install @capacitor/camera
       - GPS para verificar visitas de cobranza:
           npm install @capacitor/geolocation
       - Push notifications (recordatorios pagos vencidos):
           npm install @capacitor/push-notifications
           + configurar FCM (Android) y APNs (iOS)
       - Biometria para login rapido:
           npm install @capacitor/biometric-auth (o capacitor-native-biometric)
       - Compartir contratos PDF a WhatsApp:
           npm install @capacitor/share

    PASO 4 — PUBLICACION EN TIENDAS (1-3 dias + review)
       Android (Google Play):
         - Crear cuenta en https://play.google.com/console ($25 una vez)
         - Generar APK firmado con keystore
         - Llenar metadatos, screenshots, politica privacidad
         - Subir a "Internal testing" primero, luego "Production"
         - Review tipicamente 1-3 dias

       iOS (App Store):
         - Inscribirse en https://developer.apple.com/programs/ ($99/ano)
         - Generar certificados y provisioning profiles en Xcode
         - Subir a App Store Connect
         - Llenar metadatos, screenshots
         - Review tipicamente 1-3 dias (a veces mas estricto)

    CONSIDERACIONES:
       - VITE_API_URL en mobile debe apuntar al backend de Render
         (https://prestamax-api.onrender.com/api), no a localhost
       - CORS en backend ya permite el origen, no requiere cambios
       - Stripe Checkout funciona via WebView dentro de la app
       - Para "live updates" sin republicar (parches menores):
         npm install @capgo/capacitor-updater  (gratis hasta cierto limite)
       - El bundle se vuelve mas pesado (~3-5 MB), pero aceptable

    REQUISITOS DE HARDWARE:
       - Android: cualquier maquina con Android Studio
       - iOS: SOLO funciona en Mac con Xcode (no se puede compilar
         iOS desde Windows/Linux)
       - Si no tienes Mac: alternativas como MacInCloud o EAS Build de Expo
