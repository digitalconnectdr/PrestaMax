━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

PrestaMax es un sistema SaaS (Software as a Service) multi-tenant de gestión de
préstamos personales, diseñado para ser vendido por suscripción a prestamistas
individuales o empresas prestamistas en República Dominicana y el Caribe.

2. ARQUITECTURA MULTI-TENANT
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

El sistema opera con aislamiento total de datos entre prestamistas (tenants).
Cada empresa prestamista es un "tenant" con sus propios datos, usuarios,
configuraciones y plan de suscripción.

NIVELES DE USUARIO:
  1. platform_owner   — Dueño de PrestaMax. Acceso total al panel de administración.
                        Puede crear/gestionar tenants, planes, usuarios globales,
                        ver auditorías, hacer backups, borrar datos por empresa.
  2. platform_admin   — Administrador de plataforma. Similar al owner pero sin
                        algunas acciones destructivas.
  3. platform_support — Soporte técnico de la plataforma.
  4. tenant_owner     — Dueño de una empresa prestamista. Acceso total dentro de
                        su tenant. Puede invitar cobradores y gestionar su negocio.
  5. admin            — Administrador dentro de un tenant. Acceso completo.
  6. prestamista      — Oficial de préstamos. Puede crear, aprobar, desembolsar.
  7. oficial          — Igual que prestamista (alias).
  8. cashier          — Cajero. Principalmente registra pagos y recibos.
  9. cobrador         — Cobrador. Ve préstamos asignados, registra pagos, cobros.

UN USUARIO PUEDE PERTENECER A MÚLTIPLES TENANTS (empresas), cada membresía
tiene sus propios roles y permisos dentro de cada tenant.

CABECERAS HTTP REQUERIDAS:
  - Authorization: Bearer <JWT_TOKEN>
  - x-tenant-id: <UUID del tenant>  (requerido en todas las rutas tenant-scoped)

VERIFICACIÓN DE SUSCRIPCIÓN:
  El middleware requireTenant verifica automáticamente que la suscripción del
  tenant no haya expirado. Si expiró, responde HTTP 402 con:
  { error: "Tu suscripción ha expirado...", code: "SUBSCRIPTION_EXPIRED" }

5. BASE DE DATOS — TABLAS PRINCIPALES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

TABLAS Y SU PROPÓSITO:

  plans                   — Planes de suscripción disponibles (Starter, Básico,
                            Profesional, Enterprise, Trial). Incluyen precio,
                            límites (cobradores, clientes, usuarios) y features
                            en formato JSON array de PermKeys.

  users                   — Usuarios globales del sistema. platform_role define
                            si son dueño/admin/soporte de la plataforma o "none".

  tenants                 — Empresas prestamistas. Cada una tiene su plan,
                            estado de suscripción, configuración de moneda,
                            token público para el portal de solicitudes, y datos
                            notariales para contratos legales.

  tenant_settings         — Configuración avanzada por tenant: tasa de mora diaria,
                            días de gracia, base de mora, tipo de rebate, pesos
                            del score crediticio, configuración WhatsApp,
                            multi-moneda habilitado, monedas habilitadas.

  tenant_memberships      — Relación usuario-empresa. Un usuario puede tener
                            membresías en múltiples empresas. Cada membresía
                            almacena roles (JSON array) y permisos explícitos
                            (JSON object key:boolean).

  branches                — Sucursales de cada empresa prestamista.

  clients                 — Clientes de préstamos. Incluye datos personales,
                            laborales, referencias, score crediticio (1-5),
                            consentimientos (datos y WhatsApp).

  client_references       — Referencias personales/laborales del cliente.

  guarantors              — Fiadores (personas que garantizan préstamos).

  client_documents        — Documentos adjuntos del cliente (cédula, comprobantes).

  loan_products           — Productos de préstamo configurables: tipo, tasas,
                            plazos, frecuencias, amortización, comisiones,
                            si requiere garantía, política de rebate.

  loans                   — Préstamos. Almacena todo el ciclo de vida:
                            draft → pending_approval → approved → disbursed →
                            active → liquidated / in_mora / write_off / voided.
                            Incluye saldos (principal, intereses, mora, cargos),
                            días vencidos, cobrador asignado, moneda, tasa de
                            cambio, cargos por tiempo excedido (overtime).

  installments            — Cuotas del préstamo. Generadas al momento de
                            desembolso. Cada cuota tiene fecha de vencimiento,
                            monto capital, interés, mora acumulada, estado
                            (pending/partial/paid/overdue/waived).

  loan_guarantors         — Relación préstamo-fiador (M:M).

  loan_guarantees         — Garantías en especie del préstamo (vehículos,
                            propiedades, equipos, etc.).

  guarantee_categories    — Categorías de garantías en especie por tenant.

  payments                — Pagos registrados contra préstamos. Incluye
                            distribución del pago (mora, cargos, interés, capital),
                            método de pago, cuenta bancaria destino, tipo
                            (regular/interest_only/prorroga/prepago),
                            si fue anulado y por quién.

  payment_items           — Detalle por cuota de cómo se distribuyó cada pago.

  receipts                — Recibos emitidos por cada pago. Solo validez operativa.
                            Incluye número de serie, datos del cliente, resumen.

  receipt_series          — Series numéricas de recibos por tenant.

  contracts               — Contratos generados para préstamos. Modo de firma:
                            físico o digital. Estado: generated/signed/cancelled.

  contract_templates      — Plantillas de contratos por tenant. Incluyen variables
                            {{cliente}}, {{monto}}, {{cuotas}}, etc. Se pre-seeden
                            dos plantillas estándar: Pagaré Estándar y Pagaré Notarial.

  whatsapp_templates      — Plantillas de mensajes WhatsApp reutilizables por evento.

  whatsapp_messages       — Historial de mensajes enviados por WhatsApp.

  bank_accounts           — Cuentas bancarias del prestamista. Rastrean saldo
                            inicial, saldo actual, monto prestado.

  account_transfers       — Transferencias entre cuentas bancarias propias.

  income_expenses         — Registro de ingresos y gastos del negocio
                            (distintos a pagos de préstamos).

  payment_promises        — Promesas de pago registradas por cobradores,
                            con seguimiento de visitas.

  collection_notes        — Notas de gestión de cobranza por préstamo.

  collection_tasks        — Tareas asignadas a cobradores con prioridad, tipo,
                            fecha límite y resultado.

  notifications           — Notificaciones internas del sistema para usuarios.

  audit_logs              — Log de auditoría de todas las acciones importantes.
                            Incluye tenant, usuario, acción, entidad, valores
                            anteriores/nuevos.

  loan_requests           — Solicitudes enviadas desde el portal público.
                            Incluye datos personales del solicitante, documentos
                            de identidad en Base64, estado de revisión.

6. SISTEMA DE PERMISOS (permissions.ts)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Todos los permisos son dot-namespaced: "módulo.acción"

MÓDULOS Y PERMISOS:
  clients.*       — view, create, edit, delete
  loans.*         — view, create, edit, approve, reject, disburse, void,
                    write_off, import
  payments.*      — view, create, void, edit
  receipts.*      — view, reprint
  contracts.*     — view, create, sign, delete
  collections.*   — view, notes, promises, manage, tasks, tasks.manage
  requests.*      — view, approve, reject, convert
  reports.*       — dashboard, portfolio, mora, collections, advanced,
                    income, projection, datacredito
  whatsapp.*      — view, send, templates
  income.*        — view, create, edit, delete
  settings.*      — general, users, branches, products, bank_accounts
  templates.*     — view, create, edit, delete
  calculator.*    — use

PERMISOS POR DEFECTO POR ROL:
  tenant_owner / admin → TODOS los permisos
  prestamista / oficial / loan_officer → Permisos completos operativos
                                          (sin gestión de settings avanzada)
  cashier      → Solo pagos, recibos, ver clientes/préstamos, cobros básicos
  cobrador     → Ver clientes/préstamos asignados, registrar pagos, cobros

CAPAS DE VERIFICACIÓN (requirePermission middleware):
  1. Platform roles (owner/admin) — bypass total
  2. Plan feature gate — verifica si el plan del tenant incluye el permKey
  3. Role-based — computePermissions(roles, explicit, planFeatures)

IMPORTANTE: permissions.ts debe mantenerse SINCRONIZADO entre
  backend/src/lib/permissions.ts  y  frontend/src/lib/permissions.ts

8. TIPOS DE PRÉSTAMO
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

loan_products.type:
  personal         — Préstamo personal estándar
  san              — Préstamo tipo San (grupo solidario, es_san_type=1)
  commercial       — Préstamo comercial
  guarantee        — Préstamo con garantía (requires_guarantee=1)

TIPOS DE AMORTIZACIÓN (amortization_type):
  fixed_installment — Cuota fija (francesa): capital+interés constante
  flat_interest      — Interés plano: interés calculado sobre capital inicial
  interest_only      — Solo intereses por período, capital al final
  declining_balance  — Balance decreciente: capital fijo + intereses decrecientes

TIPOS DE TASA (rate_type):
  daily, weekly, biweekly, monthly, annual
  Se convierte a tasa mensual internamente para los cálculos.

FRECUENCIAS DE PAGO (payment_frequency):
  daily, every_2_days, weekly, biweekly, monthly

PRÉSTAMOS POR RÉDITOS (is_reditos=1):
  El sistema soporta préstamos donde se pagan solo intereses por un período
  y el capital al final. Se usa amortization_type = 'interest_only'.

CICLO DE VIDA DEL PRÉSTAMO:
  draft → pending_approval → approved → disbursed → active
  active puede transicionar a: liquidated / in_mora / write_off / voided

CÁLCULO DE MORA:
  - mora_rate_daily: tasa diaria de mora (default 0.1% = 0.001)
  - mora_grace_days: días de gracia antes de calcular mora (default 3)
  - mora_base: puede ser 'cuota_vencida' (sobre el monto de la cuota vencida)
                o 'saldo_capital' (sobre el capital pendiente)
  - mora_max_rate: tope máximo de mora (default 30%)
  - mora_fixed_enabled: si TRUE, aplica un monto fijo de mora en vez de porcentaje
  - Formula: mora_daily = base_amount × mora_rate_daily × dias_vencidos

OVERTIME (tiempo excedido):
  Si un préstamo llega a su fecha de vencimiento (maturity_date) y aún no
  fue liquidado, se calcula un recargo proporcional a los días de retraso:
  overtime_charge acumulado en loans.overtime_charge
  overtime_days acumulado en loans.overtime_days

PRORROGA (extensión de plazo):
  Un pago de tipo 'prorroga' desplaza todas las cuotas pendientes hacia adelante
  en un período de pago. Se cobra un prorroga_fee (configurable en el préstamo).
  La fecha original se preserva en due_date; la fecha desplazada en
  deferred_due_date de cada cuota.

REBATE (descuento por pago anticipado):
  Si un cliente paga la totalidad antes de la fecha acordada, se aplica rebate:
  - proportional: 100% de los intereses futuros restantes (cancelados)
  - partial:       50% de los intereses futuros restantes
  - fixed_rate:    porcentaje configurable (rebate_rate) de intereses restantes
  - none:          sin rebate

SCORE CREDITICIO (1-5):
  Calculado automáticamente en cada pago registrado o anulado.
  Factores (pesos configurables por tenant):
    - Puntualidad de pagos (default 40%)
    - Ratio de préstamos pagados (default 30%)
    - Antigüedad del cliente en meses (default 20%)
    - Sin mora histórica (default 10%)
  Formula: raw = punctuality*0.5 + paidRatio*0.3 + ageMonths*0.2
           score = max(1, min(5, round(raw * 4) + 1))
  Modos: global (un score unificado) o configurable por prestamista.

IMPORTACIÓN CSV:
  El sistema acepta importación masiva de préstamos desde CSV.
  Columnas requeridas incluyen datos del cliente, del préstamo y de cuotas.
  Los préstamos importados incluyen campos currency y exchange_rate_to_dop.

9. MÓDULO DE PAGOS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Orden de aplicación de pagos (configurable por tenant):
  Default: mora → cargos → interés → capital
  (almacenado en tenants.payment_order como JSON array)

Tipos de pago:
  regular         — Pago normal de cuotas
  interest_only   — Solo paga intereses (difiere capital a la siguiente cuota)
  prorroga        — Extiende el plazo del préstamo
  prepago         — Pago anticipado (puede activar rebate)

Anulación de pagos (void):
  Solo tenant_owner o platform_owner pueden anular pagos.
  Al anular: se recalculan todas las cuotas desde cero re-aplicando los
  pagos restantes no anulados en orden cronológico.
  Se re-calcula días_vencidos, saldos, y estado del préstamo.
  Requiere motivo de anulación.
  El recibo asociado se marca como anulado también.

Validaciones al registrar pago:
  - Monto debe ser > 0
  - El préstamo no puede estar en estado: liquidated, paid, cancelled, rejected
  - Se actualiza el score del cliente automáticamente

16. CIBERSEGURIDAD IMPLEMENTADA
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

MEDIDAS DE SEGURIDAD ACTIVAS:

  JWT (auth.ts):
    - Algoritmo forzado: { algorithms: ['HS256'] } — previene algorithm confusion
    - Token de 7 días de expiración
    - JWT_SECRET validado al inicio (error si no está configurado)

  Helmet.js:
    - CSP en producción con whitelist estricta
    - noSniff, frameguard: deny, hidePoweredBy, xssFilter
    - HSTS en producción (maxAge 1 año, includeSubDomains, preload)
    - Referrer-Policy: strict-origin-when-cross-origin

  CORS:
    - Whitelist estricta: FRONTEND_URL, prestamax.com, app.prestamax.com
    - Credentials: true, methods: GET/POST/PUT/DELETE/PATCH

  Rate Limiting (express-rate-limit):
    - Global: 200 req/15min por IP
    - Auth endpoints: 15 intentos/15min (anti-brute-force)
    - /health exento del rate limit global

  Middleware de seguridad (securityMiddleware.ts):
    - sanitizeInputs: limpia inputs HTML/XSS en req.body
    - blockKnownBots: bloquea user agents maliciosos
    - detectMaliciousPayload: detecta intentos de SQL injection y XSS
    - auditLogger: registra acciones sensibles en audit_logs

  Aislamiento de Tenants:
    - requireTenant verifica ownership de cada recurso
    - Todas las queries incluyen tenant_id en WHERE
    - IDOR corregidos: loans.reject, contracts.delete, payments.void
      todos usan AND tenant_id=? en sus queries

  Contraseñas:
    - Bcrypt con cost factor 12
    - Validación de complejidad: min 8 chars, 1 mayúscula, 1 número,
      1 carácter especial
    - Aplicado en: register-tenant, change-password, admin reset-password

  Bootstrap protegido:
    - /api/admin/bootstrap requiere requireFirstBootstrap middleware
    - Si ya existe un admin, solo admins pueden ejecutarlo

  ErrorHandler:
    - En producción: errores 5xx devuelven mensaje genérico (no internals)
    - En desarrollo: muestra mensaje real para debugging
    - Siempre loguea el stack completo en consola del servidor

  /health endpoint:
    - Solo devuelve { status: "ok" } — sin version ni nombre del sistema

PENDIENTES CONOCIDOS (no implementados):
  - Token blacklist/logout (los JWT son válidos hasta expirar aunque el usuario
    haga logout — requeriría Redis o tabla de tokens revocados)
  - Validación de MIME type/tamaño en imágenes Base64 del portal público
  - Rate limiting individual por usuario autenticado (además de por IP)

19. CONFIGURACIÓN DE LA EMPRESA PRESTAMISTA (Settings)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Desde Settings > General, el prestamista puede configurar:
  - Datos de la empresa (nombre, RNC, teléfono, email, dirección, logo, firma)
  - Representante legal y datos notariales (para contratos)
  - Testigos para contratos notariales (nombre, cédula, domicilio × 2)
  - Moneda predeterminada
  - Multi-moneda (habilitar/deshabilitar, monedas habilitadas)
  - Modo de firma (física o digital)

Configuración de mora (Settings > Mora):

  - mora_grace_days: días de gracia (ej: 3)

  - mora_base: cuota_vencida o saldo_capital
  - mora_fixed_enabled: monto fijo de mora (sí/no)
  - mora_fixed_amount: monto fijo si está habilitado

20. MIGRACIONES DE BASE DE DATOS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

No existe un sistema de migraciones formal. Las migraciones se aplican en
initializeDatabase() usando try/catch con ALTER TABLE ADD COLUMN.
Si la columna ya existe, el error se ignora silenciosamente.

Este patrón es seguro para nuevos deployments porque:
  - CREATE TABLE IF NOT EXISTS crea tablas solo si no existen
  - ALTER TABLE ADD COLUMN falla silenciosamente si la columna ya existe

Al agregar nuevas columnas, siempre usar este patrón:
  try { db.exec(`ALTER TABLE tabla ADD COLUMN nueva_col TIPO DEFAULT valor`); } catch(_) {}

22. CALCULADORA DE PRÉSTAMOS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Disponible en /calculadora. No requiere préstamo activo.

Parámetros de entrada:
  - Monto del préstamo
  - Tipo de amortización (fixed_installment, flat_interest, interest_only)
  - Tasa de interés y tipo de tasa (daily/weekly/biweekly/monthly/annual)
  - Plazo y unidad (days/weeks/months)
  - Frecuencia de pago (daily/every_2_days/weekly/biweekly/monthly)
  - Fecha del primer pago

Salida:
  - Tabla de amortización completa
  - Total a pagar, total intereses, cuota mensual
  - Gráfica de distribución capital vs interés

Conversiones de tasa interna:
  daily → mensual: tasa × 30
  weekly → mensual: tasa × 4.33
  biweekly → mensual: tasa × 2
  annual → mensual: tasa / 12
  monthly → directa

23. NOTAS IMPORTANTES PARA DESARROLLO FUTURO
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. SINCRONIZACIÓN DE PERMISOS: Cualquier cambio en backend/src/lib/permissions.ts
   DEBE replicarse manualmente en frontend/src/lib/permissions.ts.
   Estos dos archivos deben ser idénticos.

2. MIGRACIONES: Siempre usar el patrón try/catch con ALTER TABLE ADD COLUMN
   en la función initializeDatabase() de database.ts.

3. IDOR PROTECTION: Toda query UPDATE/DELETE debe incluir AND tenant_id=?
   para prevenir acceso cruzado entre tenants.

4. SCORE: El score se almacena en la tabla clients (score INTEGER 1-5).
   Se recalcula automáticamente en recalcClientScore() después de cada pago
   o anulación. La escala es 1-5 (no 0-100).

5. SALDO TOTAL: total_balance = principal_balance + interest_balance +
                                mora_balance + charges_balance

6. DÍAS VENCIDOS: Se recalcula usando SQL MIN(julianday('now') - julianday(due_date))
   sobre cuotas en estado pending/partial con due_date < hoy.

7. CONTRASEÑAS: Las contraseñas del propietario (jcpenalo@gmail.com) fueron
   configuradas directamente en la DB (no por API), por lo que no pasan
   las validaciones de complejidad que sí aplican a usuarios nuevos.

8. WHATSAPP: No hay integración real con la API de WhatsApp. El sistema registra
   los mensajes como 'sent' pero no hay implementación real de envío. Se necesita
   integrar con Twilio, Meta WhatsApp Business API u otra solución.

9. STRIPE: La integración con Stripe está parcialmente implementada. Las tablas
   tienen campos stripe_customer_id y stripe_subscription_id pero la lógica
   completa de webhooks y cobro automático puede requerir trabajo adicional.

10. AUTO-SEED: Si la DB está vacía al iniciar (users count = 0), el sistema
    intenta cargar datos de demo desde seed.ts. En producción esto no debe
    ocurrir porque ya existe el usuario del platform_owner.

11. TENANT REQUERIDO PARA OPERAR: El platform_owner también necesita pertenecer
    a al menos un tenant para que el frontend envíe el header x-tenant-id y las
    rutas de operaciones (dashboard, préstamos, pagos, etc.) funcionen. Sin tenant,
    el frontend recibe tenants=[] en el login y todas las llamadas tenant-scoped
    devuelven 400 "Tenant ID requerido".
    SOLUCIÓN APLICADA: Se creó el tenant "PrestaMax Demo" para jcpenalo@gmail.com.

12. DASHBOARD FALLBACK: DashboardPage.tsx tenía datos demo hardcodeados (75,000,
    5 clientes, etc.) que se mostraban cuando la API fallaba. Fue corregido para
    mostrar ceros en caso de error, evitando confusión con datos ficticios.

13. ESTADO INICIAL DE LA BD (post-reset Abril 2026):
    - 1 usuario: jcpenalo@gmail.com (platform_owner)
    - 1 tenant: "PrestaMax Demo" (plan Enterprise, suscripción hasta 2099)
    - 6 planes: Plan Trial, Starter, Básico, Profesional, Enterprise, Empresarial
    - 0 clientes, 0 préstamos, 0 pagos (sistema limpio)
